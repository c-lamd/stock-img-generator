    **1. print_dry_run_table(tasks_by_template):**

    - Print tabulated table using tabulate with headers ["Template", "Combo", "Prompt (truncated)"]

    - Per Pattern 2 from RESEARCH.md

    **2. run_template_preview_loop(provider_name, api_key, templates, demographics, output_dir, size_value):**
    - For each template in templates:
      - Take first demographic combo (first ethnicity, first age, first gender)
      - Call expand_to_tasks with single template, single demo combo, output_dir / "_preview"
      - Print "Generating preview for: {tmpl.name}"
      - Call asyncio.run(generate_batch(provider_name, api_key, [preview_task], size_value))
      - Print saved path
      - questionary.select with choices: "Approve", "Skip", "Abort"
      - If Abort: sys.exit(0)
      - If Approve: add to approved list
      - If Skip: continue
    - Return list of approved templates
    - Per Pattern 4 from RESEARCH.md
    - IMPORTANT: Do NOT refactor to asyncio.gather — sequential by design (per Pitfall 1 and Anti-Pattern from RESEARCH.md)

    **3. Complete main() flow:**
    Wire the full template-based workflow. The flow should be:
    a. Parse args: --output (default "output"), --no-preview, --dry-run
    b. load_dotenv()
    c. select_provider(), select_resolution()
    d. select_templates(), select_demographics()
    e. For each template, apply_demographic_restrictions, then expand_to_tasks — collect all tasks and build tasks_by_template dict for dry-run display
    f. If --dry-run: call print_dry_run_table(tasks_by_template) and exit (no API calls)
    g. Build template_task_counts list for cost confirmation
    h. show_cost_confirmation() — show BEFORE preview (per Pitfall 2 from RESEARCH.md: cost gate fires before spending money on previews)
    i. If not --no-preview: run_template_preview_loop() — may reduce template list
    j. If templates were filtered by preview, rebuild tasks for approved templates only
    k. asyncio.run(generate_batch()) with all tasks
    l. print_failure_report() if failures (note: need to handle the fact that generate_batch returns failure count, not the failed list — see note below)
    m. Print completion summary

    **Handling generate_batch failures:**
    providers.py's generate_batch() returns int (failure count) and prints failures to stdout internally. Since providers.py is LOCKED, we cannot modify it to return the failed list. Two options:
    - Option A: Accept that providers.py already prints basic failure info (it prints up to 10 failures with path and error). Add print_failure_report as a post-processing step that checks which output_path files do NOT exist after generation — those are failures. For content policy classification, we would need the error string which is not returned.
    - Option B (recommended): After the batch, simply print a summary line "X images failed" using the returned count. The detailed classify_failure / print_failure_report functions from Plan 01 become available when/if generate_batch is later updated to return the failed list. For now, add a comment noting this limitation.

    Use option B: rely on providers.py's built-in failure reporting (it already prints path + error for each failure). The classify_failure and print_failure_report functions exist and are tested, ready for use when the failure list becomes accessible. Add a TODO comment noting this.
